# Entity Links Minjust Batch

Массовый поиск официальных сайтов и социальных страниц для людей и
организаций. Проект рассчитан на большой CSV (например, 22 000 сущностей) и
бережно расходует бесплатные grounded-запросы Gemini.

## Как работает новый каскад

1. Входной CSV нормализуется и распределяется по стабильным шардам.
2. Проверяется SQLite-кэш.
3. Официальный REST-реестр Минюста скачивается блоками через `offset/limit`.
4. Имена из входного CSV сопоставляются с реестром локально, без LLM.
5. Ссылки берутся напрямую из `field_6_s` официального ответа Минюста.
6. Только нерешённые сущности собираются в пакеты по 5.
7. Один grounded prompt Gemini обрабатывает весь пакет, сохраняя `entity_id`.
8. Каждый результат сохраняется отдельно по сущности.
9. Суточное использование Gemini хранится в SQLite и не сбрасывается при
   перезапуске программы.

## Бесплатный режим

По умолчанию используется:

```env
GEMINI_MODEL=gemini-2.5-flash-lite
LLM_BATCH_SIZE=5
LLM_DAILY_REQUEST_BUDGET=480
LLM_BUDGET_TIMEZONE=America/Los_Angeles
```

Официальная бесплатная квота Grounding with Google Search для Gemini 2.5
Flash и Flash-Lite составляет до 500 grounded prompts в сутки и является
общей для этих двух моделей. В проекте установлен лимит 480, чтобы оставить
резерв для ручных тестов и других приложений.

При пакете из пяти один день позволяет отправить до 2400 сущностей. Фактическая
скорость зависит от того, сколько записей уже закроет Минюст, качества ответов
модели и действующих лимитов конкретного проекта в Google AI Studio.

## Установка

Рекомендуется Python 3.12.

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt
```

Открой `.env` и заполни только токен:

```env
GEMINI_API_KEY=твой_ключ
```

Все остальные параметры уже заполнены.

## Входной CSV

Путь задаётся в `.env`:

```env
INPUT_CSV=data/entities.csv
```

Формат:

```csv
id,name,entity_type
1,Дудь Юрий Александрович,person
2,The Moscow Times,organization
3,Название организации,organization
```

Допустимые типы:

- `person`
- `organization`
- `unknown`

Колонка `id` желательна. При её отсутствии используется номер строки.

## Запуск

```powershell
python main.py
```

Аргументов командной строки нет. Настройки берутся из `.env`.

## Выходные данные

По умолчанию создаются:

```text
data/entity_links.sqlite3
data/results.jsonl
data/unresolved.csv
```

SQLite содержит:

- кэш результатов по сущностям;
- локальную копию официального реестра Минюста;
- исходный JSON каждой записи Минюста;
- разобранные ссылки;
- постоянный суточный счётчик LLM-запросов.

Таблица использования LLM называется `llm_daily_usage` и содержит:

- дату в Pacific Time;
- модель;
- число фактически отправленных запросов;
- число сущностей в пакетах;
- число успешно возвращённых сущностей;
- число неудачных запросов.

## Продолжение после остановки

Просто снова запусти:

```powershell
python main.py
```

Успешные сущности берутся из SQLite. Суточный счётчик также читается из
SQLite, поэтому перезапуск программы не выдаёт новую квоту.

Когда исчерпан `LLM_DAILY_REQUEST_BUDGET`, оставшиеся сущности получают статус
`unresolved` без увеличения числа попыток. После сброса суточной квоты следующий
запуск продолжит их обработку.

## Повторные попытки

```env
LLM_MAX_ATTEMPTS=2
MAX_LLM_ATTEMPTS_PER_ENTITY=2
```

`LLM_MAX_ATTEMPTS` — повторы одного пакетного HTTP/API-вызова только при
временных ошибках (`429`, `500`, `502`, `503`, `504`). Каждый реально
отправленный повтор учитывается в суточной квоте.

`MAX_LLM_ATTEMPTS_PER_ENTITY` — сколько отдельных запусков поиска разрешено для
сущности, если модель вернула пустой или неполный результат. Это защищает от
вечного расходования квоты на заведомо нерешаемые имена.

## Почему результаты не смешиваются

В пакет модели передаётся массив вида:

```json
[
  {
    "entity_id": "уникальный-ключ-1",
    "name": "Первое имя",
    "entity_type": "person"
  },
  {
    "entity_id": "уникальный-ключ-2",
    "name": "Вторая организация",
    "entity_type": "organization"
  }
]
```

Модель обязана вернуть отдельный объект на каждый `entity_id`. Программа:

- игнорирует неизвестные ID;
- удаляет дубликаты ID;
- сохраняет каждый результат отдельно;
- повторно ставит в очередь только отсутствующие сущности.

## Режимы обработки

### economy

```env
RESOLUTION_MODE=economy
```

Gemini вызывается только когда реестр Минюста не дал ни одной ссылки.
Рекомендуемый режим для 22 000 записей.

### balanced

```env
RESOLUTION_MODE=balanced
MIN_MINJUST_LINKS_TO_SKIP_LLM=2
```

Gemini вызывается, пока Минюст не дал требуемое число ссылок.

### full

```env
RESOLUTION_MODE=full
```

Gemini вызывается для всех сущностей. Этот режим быстро расходует квоту.

## Обновление реестра Минюста

Автоматически реестр обновляется раз в 24 часа:

```env
MINJUST_REGISTRY_TTL_HOURS=24
```

Принудительное обновление:

```powershell
python rebuild_minjust_registry.py
```

Новая копия сначала полностью загружается и проверяется. Старая таблица
заменяется одной транзакцией только после успешного скачивания всех страниц.
Если API временно недоступен, используется последняя локальная копия.

## Распределение на несколько процессов

Для четырёх независимых частей используй одинаковый CSV и разные настройки:

```env
SHARD_COUNT=4
SHARD_INDEX=0
```

Другие процессы получают `SHARD_INDEX=1`, `2`, `3`.

Для каждого шарда автоматически создаются отдельные файлы:

```text
entity_links.shard-0-of-4.sqlite3
results.shard-0-of-4.jsonl
unresolved.shard-0-of-4.csv
```

Важно: бесплатный RPD применяется к Google Cloud-проекту, а не к API-ключу.
Несколько шардов с ключами одного проекта используют общую внешнюю квоту.
Локальный SQLite-счётчик не координируется между разными машинами, поэтому при
распределённом запуске раздели `LLM_DAILY_REQUEST_BUDGET` между процессами или
используй общую PostgreSQL-базу.
